Architecture Proposals · 2026-04-22

Anneal

Controlled heating, slow cooling, and iterative tempering — applied to work plans.

Anneal is a Claude Code plugin that converts a vague task into a rigorously-reviewed execution artifact: an XML prompt, a plan directory, and the skill enrichment needed to run it. It replaces the earlier deepest-plan prototype (shipped with 91 validator defects due to asymmetric vendoring) with a cleaner core built around three named plan-review archetypes and an always-on red team.

This document proposes three distinct architectures. Each satisfies the same invariants — red team always runs, validation always runs, output is always an XML prompt and a plan directory, always enriched with the user's skills — but they generate the plan itself through three different physical analogs: single-pour Cast, blended Alloy, and iterative Temper.

Plugin anneal · v0.1.0 Output ~/Desktop/anneal/ Replaces deepest-plan v1 Inspires Metis / Momus / Oracle roles borrowed from oh-my-openagent
Prior art — oh-my-openagent Anneal's reviewer archetypes are directly inspired by oh-my-openagent's Greek-god agent taxonomy (github.com/code-yeongyu/oh-my-openagent). Metis, Momus, and Oracle are adapted from their system. Their 16-agent pre-publish-review pattern (10 per-change deep dives + 5 holistic + 1 synthesis, all launched in parallel with run_in_background=true) is the source of anneal's unbounded verify-loop discipline. Verdict tiers (SAFE / CAUTION / RISKY / BLOCK) are borrowed wholesale.

  • Red team · non-negotiableThree parallel adversaries run on every plan attempt. They are not a flag. You cannot opt out.
  • Validate · non-negotiableFunctional validation (no mocks, no test files) on a real runnable artifact before emission.
  • Dual output · XML + planEvery run produces an Opus 4.7 semantic-XML prompt and a plan directory. Not one. Both.

Section II

The Roster

Seven named agents. Each has a single purpose and a Greek etymology that signals when they run. Mix-and-match across the three proposals.

Pre-plan consultant
Metis
Greek: goddess of wisdom, prudence, deep counsel.

Reads the user's task before planning begins. Catches ambiguity, unstated requirements, AI-slop patterns (over-engineering, scope creep). Returns clarifying questions and directives for whatever planner runs next.

Runs: Always, Stage 3 (Enrich)
Planner
Prometheus
Titan of forethought; stole fire for craft.

The actual plan-writer. Interview mode for scoping, then produces a markdown plan with task graph, dependencies, verification criteria. Plans only — never touches code. Forbidden paths enforced by hook.

Runs: Stage 4. Count: 1 in Cast · N in Alloy · 1 per depth in Temper
Post-plan reviewer
Momus
Greek god of satire; mocked even the works of gods.

Ruthless adversarial audit of the finished plan. Finds every gap, ambiguity, missing context that would block implementation. Outputs verdict and blocking-issue list with file/line citations where applicable.

Runs: Stage 4 close-out
Always-on red team
Red-Team Trinity
Three parallel adversaries: Security · Scope · Assumptions.

Three background agents poke the plan from different angles. Never sequential. Each owns a distinct failure mode. At least one FAIL from any of the three routes the plan back to Planner with the failure as an added constraint.

Runs: Stage 5. Parallel: 3 always
Architecture synthesizer
Oracle
Delphic seer; reads architecture whole, not piecewise.

Bird's-eye review after Red Team. Catches what per-piece reviewers miss: coherence, release risk, version-bump semantics, migration cost. Emits verdict tier: SAFE / CAUTION / RISKY / BLOCK.

Runs: Stage 5 close-out
Functional validator
Hephaestus
Craftsman of the gods; tests by building.

Runs the plan against a real artifact. No mocks, no test files, no stubs. Builds, runs, captures evidence (screenshots, logs, API responses). If evidence is empty or the artifact doesn't boot, verdict is FAIL.

Runs: Stage 6
Emitter
Atlas
Bearer of the world; carries the whole artifact forward.

Assembles the final XML prompt (Opus 4.7 semantic tags) and the plan directory. Enforces output schema. Writes to ~/Desktop/anneal-runs/{slug}/. This is the only agent permitted to write outside the plan's scoped directory.

Runs: Stage 7
Proposal A
I
Architecture A · Linear

Cast

Single-pour. One planner. One red team. One oracle. One validate. Emit.
Cost profile
~8 agent spawns

A single, disciplined pour through the mould. The fastest path from task to artifact. Red team still runs — always — but the plan itself is produced by one planner in one pass. Good for tasks where the scope is clear and correctness matters more than exploration.

Cmd+wheel zoom · drag to pan · double-click to fit

Loading

Strengths

  • Cheapest. ~8 agent spawns per successful run.
  • Fastest wall-clock. Single planner, no tournament overhead.
  • Full invariant coverage. Red team + Oracle + validate all present.
  • Re-loop on FAIL is deterministic — route back to Planner with failure as input.

Weaknesses

  • No exploration of alternative plan shapes. If the first planner misses a creative approach, nobody else will find it.
  • Re-loops can diverge — each retry is a new independent plan, not a refinement of the prior attempt.
  • Best fit: well-scoped tasks, bug fixes, refactors. Not for novel architecture.

Default CLI

  • /anneal <task> --fast invokes Cast.
  • Implicit default when --versions and --deepen are both absent.
Proposal B
II
Architecture B · Tournament

Alloy

N parallel planners compete. A synthesizer blends their best material into one plan.
Cost profile
~8 + 2N spawns

Five planners, five approaches, one blend. The user says --versions 5 and five planners run in parallel, each with a different prompt-shape bias. A synthesizer (the Alloy itself) merges the strongest elements. Expensive, but catches the creative approach that no single planner would have found alone.

Cmd+wheel zoom · drag to pan · double-click to fit

Loading

Strengths

  • Highest plan quality. Five independent interpretations surface more of the problem space than one.
  • Synthesis is where the creative leaps happen — blending a defensive plan with a UX plan often produces something neither would have written alone.
  • Natural fit for the user's request: --versions 5 then consolidate.
  • Planners run concurrently; wall-clock ~= one planner's time.

Weaknesses

  • Most expensive. 5× the planning cost per run.
  • Synthesizer is itself a single point of failure. If it blends poorly, Momus + Red Team have to catch it.
  • Best fit: novel architecture, greenfield features, high-stakes refactors.

Default CLI

  • /anneal <task> --versions 5 invokes Alloy with 5 planners.
  • --versions 3 for a cheaper blend, --versions 7 for a richer one.
Proposal C
III
Architecture C · Fixed-Point

Temper

One plan, heated and cooled repeatedly until variance of top-3 scores settles below 0.3.
Cost profile
~8 × depth (1–3)

Start with a seed plan. Heat (planner rewrites aggressively). Cool (Momus + Red Team score). Heat again with the feedback baked in. Stop when scores stabilize — variance of top-3 depth scores < 0.3, or |Δ| < 0.15, or depth hits 3. The red team is not a separate stage here; it runs inside every iteration. Fixed-point convergence, not a finite pipeline.

Cmd+wheel zoom · drag to pan · double-click to fit

Loading

Strengths

  • Red team runs every iteration — genuinely always-on inside the loop.
  • Convergence is principled, not arbitrary. Variance-based stopping rule means the plan stops refining when it truly stops improving.
  • Each iteration carries prior feedback forward. Unlike Cast's re-loop, Temper actually learns.
  • Caps depth at 3 so runaway iteration is impossible.

Weaknesses

  • Cost scales with depth. Worst case 3× Cast.
  • Convergence math is subtle; bugs in the scoring rubric can cause early or late termination.
  • Best fit: complex but well-scoped tasks. The plugin rewrite itself is the ideal test case.

Default CLI

  • /anneal <task> --deepen invokes Temper with default depth cap 3.
  • --depth 2 for cheaper, --depth 5 if you really want to melt the rock.
  • --ultra combines Alloy + Temper: N-way tournament, then deepen the winner.
Section III

Worked Example · The Plugin Rewrite

Same task, three paths. How each architecture would approach the canonical bootstrap problem: rewriting the plugin that produces anneal itself.

User input /anneal "rewrite the deepest-plan plugin to use SADD primitives and fix the 91 asymmetric-vendoring defects. ship a functional v0.1.0 that passes its own red team."

Cast Linear

One planner. Direct path. If the first plan is already close to right, this finishes first.

  1. Metis reads the task. Flags: "91 defects — which ones?" Returns 4 clarifying questions for Prometheus.
  2. Prometheus writes a single 12-phase rewrite plan. Phases 00-11 for scaffold → vendor → router → validate → ship.
  3. Momus audits: finds 6 gaps (e.g., "no rollback plan if Phase 04 fails mid-vendor"). Plan revised in place.
  4. Red-Team Trinity runs. Security finds nothing. Scope flags "Phase 11 too broad." Assumptions flags "assumes Opus 4.7 XML spec is stable."
  5. Oracle: CAUTION — "safe to ship but expect re-work on Phase 11."
  6. Hephaestus builds a dry-run. Validator confirms plugin boots, /anneal command registers, no crash on empty input.
  7. Atlas emits DEEPEST-PROMPT.xml + plan/phase-00...11.md to ~/Desktop/anneal-runs/plugin-rewrite/.
Outcome ~8 agent spawns. ~4 min wall-clock. Plan is solid but linear — no alternative architectures explored. Validate catches anything Cast's single planner missed.

Alloy Tournament

Five planners race. Each sees the same Metis directives but with a different bias — the synthesizer picks the strongest material from each.

  1. Metis returns directives. Intent Gate spawns 5 planners in parallel.
  2. P1 (correctness): phased rollout with gate tests. P2 (minimalist): 4 phases, strip ceremony. P3 (defensive): every phase has a rollback. P4 (perf): vendor only what's used. P5 (UX): add status-line progress indicator.
  3. Synthesizer reads all 5. Takes P1's phased structure, P3's rollback discipline, P5's status-line. Rejects P2's minimalism as under-scoped, P4's perf work as premature.
  4. Momus audits the blend. Finds 2 integration gaps between P1 and P3 (rollback trigger semantics). Fixed.
  5. Red-Team Trinity + Oracle as in Cast. Oracle: SAFE.
  6. Hephaestus validates. Atlas emits.
Outcome ~18 agent spawns. ~6 min wall-clock (planners parallel). Plan is richer than any single planner would have produced. Worth the cost for architecturally novel tasks.

Temper Fixed-Point

One planner, three depths. Each depth folds Red Team's feedback back into the next rewrite. Converges when scores stabilize.

  1. Metis + Seed planner produce v1. Momus score: 62/100. Red Team flags 5 issues.
  2. Depth 1: Planner rewrites with v1 score + 5 issues as constraints. Produces v2. Score: 78. Issues: 2.
  3. Depth 2: Planner rewrites v2 with feedback. Produces v3. Score: 85. Issues: 0. Variance of top-3 depth scores = 0.18 — converged.
  4. Oracle reviews v3. SAFE.
  5. Hephaestus validates. Atlas emits.
Outcome ~24 agent spawns (3 depths × 8). ~7 min wall-clock (sequential depths). Plan is the most refined of the three; each iteration genuinely improves on the prior. Ideal for the plugin rewrite because the spec has been re-scoped twice already — iteration is the bottleneck, not exploration.

Section V

CLI Contract

One command, semantic flags. The user names the task shape; the harness picks the architecture and the models.

The five forms

Pick one. All five satisfy every invariant — red team, validate, XML+plan output, skill enrichment.

# Bare · smart mode (Cast)
/anneal "fix the session-timeout bug in the auth middleware"

# Explicit fast mode (Cast)
/anneal "fix the session-timeout bug" --fast

# Tournament mode (Alloy)
/anneal "design an auth provider abstraction layer" --versions 5

# Deepen mode (Temper)
/anneal "rewrite the query optimizer to use cost-based planning" --deepen

# Ultra · Alloy then Temper the winner
/anneal "architect the multi-tenant rewrite" --ultra

# Stand-alone subcommands (same invariants)
/anneal consolidate ./candidates/        # synthesize existing plans
/anneal red-team   ./plan.md             # run Red-Team Trinity on an existing plan
/anneal validate   ./plan.md             # run Hephaestus on an existing plan
/anneal loop       "task"                 # unbounded re-loop (infinite until PASS)

Flag reference

--fastCast architecture. Single planner, one pass.
--versions NAlloy architecture with N parallel planners (default 3, range 2–7).
--deepenTemper architecture. Iterative until variance-converged.
--depth NOverride Temper depth cap (default 3, range 1–5).
--ultraAlloy (N=5) then Temper the winner. Maximum cost, maximum quality.
--loopUnbounded re-loop: re-run until PASS. No cap. Use with care.
--type <cat>Category: ultrabrain · deep · quick. Maps to model at runtime.
--no-red-teamREFUSED. Red team is non-negotiable. Flag is logged and ignored.
--no-validateREFUSED. Validation is non-negotiable. Flag is logged and ignored.
